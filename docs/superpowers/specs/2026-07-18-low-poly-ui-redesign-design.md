# BOOK돋움 질문도감 로우폴리 UI 리디자인 설계

- 작성일: 2026-07-18
- 대상 저장소: `gud8238/mokpo-imsung-app`
- 배포 대상: Netlify `mokpo-imsung-v1`
- 디자인 방향: B안 — 하이브리드 스토리북

## 1. 목표

기존 목포임성초 독서 질문 웹앱의 인증, Supabase 데이터 처리, Gemini 질문 분석, 교사 피드백, 학생 성장 분석, HWPX 다운로드 기능을 변경하지 않고 모든 페이지의 UI를 로우폴리 스타일로 업그레이드한다.

첨부된 로우폴리 자연환경 이미지의 특징인 다면체 지형, 선명한 자연색, 입체적인 책·나무 오브젝트와 깊이감을 앱의 시각 언어로 재해석한다. 배경 모션과 3D 요소는 분위기를 만들되 텍스트, 버튼, 입력창, 차트와 겹치거나 가독성을 낮추지 않아야 한다.

## 2. 확정된 디자인 방향

선택된 B안은 실시간 3D를 화면 전체에 과도하게 적용하지 않고 다음 세 요소를 혼합한다.

1. 정적 로우폴리 일러스트와 CSS 폴리곤 배경
2. 로그인 및 주요 히어로 영역의 제한적인 3D 책·나무 오브젝트
3. 높은 불투명도와 명확한 대비를 가진 전경 콘텐츠 카드

이 방식은 몰입형 3D 월드보다 로딩과 GPU 비용이 낮고, CSS 중심의 절제형 UI보다 로우폴리 세계관을 명확하게 전달한다.

## 3. 디자인 원칙

### 3.1 교육 콘텐츠 우선

- 배경은 장식이며 읽기·입력·클릭을 방해하지 않는다.
- 텍스트와 인터랙션은 불투명도 약 94% 이상의 밝은 패널 위에 배치한다.
- 배경 캔버스와 장식 레이어는 `pointer-events: none`으로 처리한다.
- 질문 유형의 고정 색상은 유지한다.
  - `factual`: 파랑
  - `inferential`: 보라
  - `evaluative`: 주황

### 3.2 영역별 역할 구분

- 학생 영역: 보라색, 라일락, 초록색을 중심으로 한 생각의 숲
- 교사 영역: 파랑색, 청회색, 차분한 보라색을 중심으로 한 질문 관찰소
- 로그인: 학생과 교사가 함께 들어오는 따뜻한 숲의 입구

### 3.3 초등학생 친화성

- 모든 사용자 문구는 한국어 해요체를 유지한다.
- 버튼과 제목은 짧고 행동 중심으로 작성한다.
- 아이콘과 3D 오브젝트는 장식보다 의미 전달을 우선한다.
- 실패·오류·AI 피드백 문구의 기존 격려 중심 어조를 유지한다.

## 4. 화면 계층 구조

각 페이지는 다음 세 층으로 구성한다.

### 4.1 배경층

- 고정 또는 절대 위치의 로우폴리 배경
- 저속 CSS 폴리곤 이동
- 로그인 및 주요 히어로의 제한적인 WebGL 장면
- 사용자 입력이나 포인터 이벤트를 받지 않음

### 4.2 분위기층

- 학생·교사 역할에 맞는 색조 오버레이
- 모든 페이지에 학생·교사 역할별 명도 조절 그라데이션 적용
- 장식용 광원, 안개, 작은 폴리곤 파편

### 4.3 콘텐츠층

- 내비게이션, 입력 폼, 카드, 차트, 모달, 알림
- 높은 명도 대비와 불투명 배경
- 키보드 포커스 및 터치 영역 유지
- 기존 서버·클라이언트 컴포넌트 경계를 유지

## 5. 페이지별 콘셉트

### 5.1 로그인

콘셉트는 “생각의 숲으로 들어가는 입구”이다.

- 가장 풍부한 로우폴리 환경과 저속 3D 모션 사용
- 로그인 카드는 화면 중앙에 배치하고 배경에서 확실히 분리
- 학교 로고와 앱 이름은 배경 위에서도 읽히도록 명암 보호
- 기존 이메일·비밀번호 입력과 로그인 서버 액션 유지
- 기존 영상 배경은 새 하이브리드 로우폴리 장면으로 교체하고, WebGL을 사용할 수 없으면 정적 로우폴리 배경을 표시

### 5.2 학생 책 목록

콘셉트는 “책이 놓인 탐험길”이다.

- 책 표지는 입체적인 탐험 오브젝트처럼 표현
- 책 카드의 제목, 저자와 선택 행동은 밝은 카드 위에 표시
- 배경에는 라일락 암석, 초록 지형과 작은 부유 폴리곤 사용
- 모바일에서는 한 열 또는 두 열로 자연스럽게 축소

### 5.3 학생 질문 작성

콘셉트는 “질문 씨앗을 키우는 숲”이다.

- 사실·궁금·라면 질문 선택 카드를 고정 색상으로 강조
- 선택, 작성, AI 분석, 결과 확인의 단계가 시각적으로 이어지도록 구성
- AI 로딩과 결과 등장은 부드러운 전환만 사용
- 입력창과 결과 패널은 장식 배경과 분리

### 5.4 학생 내 기록

콘셉트는 “독서 성장 지도”이다.

- 질문 기록을 책별 또는 시간 흐름에 따라 탐험 기록처럼 구성
- AI 피드백과 교사 피드백은 서로 다른 색의 레이어로 구분
- 빈 상태에는 로우폴리 지도·책 오브젝트와 다음 행동 안내 제공

### 5.5 교사 대시보드

콘셉트는 “전체 활동을 보는 관찰소”이다.

- 통계 카드와 차트는 고대비 패널에 배치
- 학생별·책별 활동으로 이동하는 기존 상호작용 유지
- 장식은 차트 축, 범례, 숫자와 겹치지 않는 외곽에 제한
- 교사 영역의 청색 계열을 유지해 학생 영역과 구분

### 5.6 교사 책·학생 상세

콘셉트는 “질문을 정리한 로우폴리 아카이브”이다.

- 질문, AI 피드백, 교사 피드백의 위계를 카드 깊이와 색으로 표현
- 삭제, 피드백 작성, AI 진단, HWPX 다운로드 기능 유지
- 모달과 아코디언은 배경 효과보다 내용 가독성을 우선

## 6. 공통 컴포넌트 경계

### 6.1 `LowPolyBackdrop`

- 학생·교사·로그인 변형을 지원하는 공통 배경
- CSS 폴리곤과 정적 이미지 폴백을 기본으로 제공
- 브라우저 기능과 사용자 모션 설정에 따라 3D 장면 활성화 여부 결정

### 6.2 `LowPolyScene`

- 브라우저 전용으로 동적 로드
- 로그인 및 지정된 히어로 영역에서만 렌더링
- 투명 캔버스, 제한된 오브젝트 수, 제한된 픽셀비 사용
- 리사이즈, 탭 비활성화, 언마운트 시 정지·정리 처리

### 6.3 `StorySurface`

- 카드, 패널, 모달의 공통 시각 스타일
- 약 94% 이상의 배경 불투명도
- 역할별 테두리·그림자·하이라이트 변형
- 키보드 포커스 링과 고대비 텍스트 유지

### 6.4 `LowPolyIcon` 및 로컬 자산

- 책, 질문, 학생, 교사, 기록, 대시보드 의미를 나타내는 통일된 로우폴리 아이콘
- 생성된 이미지와 아이콘은 압축된 로컬 자산으로 저장
- 앱 실행 중 외부 생성 서비스에 의존하지 않음
- 장식 이미지는 빈 `alt`, 의미 있는 이미지는 설명형 `alt` 제공

## 7. 모션 설계

### 7.1 배경 모션

- 폴리곤 이동 주기: 약 20~40초
- 3D 책·나무의 부유 주기: 약 6~10초
- 변형과 투명도 중심으로 애니메이션
- 화면이 보이지 않을 때 렌더링 중지

### 7.2 인터랙션 모션

- 버튼·카드 반응: 약 160~240ms
- 페이지 및 결과 등장: 짧은 투명도·수직 이동
- 중요한 작업 버튼에는 과도한 회전·튕김 효과를 사용하지 않음

### 7.3 모션 감소

- Mantine의 `respectReducedMotion` 활성화
- Motion 전역 설정에서 사용자 모션 감소 설정 존중
- CSS의 `prefers-reduced-motion` 대응
- 모션 감소 시 지속 회전·부유·시차 효과 중지

## 8. 성능 예산과 폴백

- 3D 코드는 초기 서버 렌더링에서 제외하고 별도 클라이언트 번들로 지연 로드한다.
- 본문 콘텐츠 표시가 3D 초기화를 기다리지 않도록 한다.
- 렌더러 픽셀비는 최대 약 1.5로 제한한다.
- 오브젝트와 광원 수를 제한하고 고비용 후처리를 사용하지 않는다.
- WebGL 초기화 실패 시 같은 색상의 정적 로우폴리 일러스트를 표시한다.
- 모바일 및 저사양 환경에서는 정적 또는 CSS 중심 배경을 우선할 수 있다.
- 생성 자산은 WebP 등 웹 친화 형식으로 압축하고 불필요한 대형 원본은 배포 번들에 포함하지 않는다.

## 9. 접근성

- 텍스트와 배경은 WCAG AA 수준의 명도 대비를 목표로 한다.
- 키보드 포커스 링을 숨기지 않는다.
- 주요 터치 대상은 최소 44px 수준을 확보한다.
- 캔버스와 장식 레이어는 보조 기술에서 의미 없는 장식으로 처리한다.
- 색상만으로 질문 유형과 상태를 전달하지 않고 라벨·아이콘을 함께 사용한다.
- 반응형 레이아웃에서 텍스트와 주요 버튼이 잘리거나 배경 뒤에 숨지 않도록 한다.

## 10. 데이터 흐름과 백엔드 범위

이번 작업은 UI 리디자인이므로 다음 데이터 흐름을 변경하지 않는다.

- Supabase 인증 및 이메일 prefix 기반 역할 라우팅
- `profiles`, `books`, `questions`, `teacher_feedbacks` 테이블 사용
- Gemini 질문 분석과 학생 성장 분석
- 질문 제출·삭제 및 교사 피드백 서버 액션
- HWPX 다운로드 API

Supabase MCP는 다음 원칙으로 연결한다.

- 프로젝트 ref `wglnrealznvbvybcoaja`로 범위 제한
- `read_only=true` 사용
- 스키마와 데이터 관계 확인 용도로만 사용
- UI 작업에 불필요한 DB 쓰기나 마이그레이션은 수행하지 않음

## 11. 오류 처리

- 3D 초기화 또는 자산 로딩 실패는 페이지 오류로 전파하지 않고 정적 배경으로 대체한다.
- 장식 자산 누락 시 기본 로우폴리 도형 또는 기존 아이콘으로 대체한다.
- 기존 Supabase·Gemini 오류 메시지와 폴백 동작을 유지한다.
- 로딩, 빈 상태, 오류 상태 모두 새 디자인 시스템으로 표현한다.
- 배경 애니메이션 오류가 폼 제출, 내비게이션, 차트 또는 모달 동작에 영향을 주지 않도록 컴포넌트를 격리한다.

### 12.1 정적 검증

- TypeScript 및 ESLint 검사
- Next.js 프로덕션 빌드
- 불필요한 클라이언트 경계 확대 여부 확인
- 생성 자산 크기와 번들 영향 확인

### 12.2 시각 검증

- 로그인, 학생 책 목록, 질문 작성, 내 기록, 교사 대시보드, 책 상세, 학생 상세 확인
- 데스크톱과 모바일 뷰포트 비교
- 배경 위 텍스트·입력·버튼·차트 가독성 확인
- 로딩·빈 상태·모달·아코디언 확인
- 모션 감소 환경과 WebGL 폴백 확인

### 12.3 기능 회귀 검증

- 로그인 및 역할별 리다이렉트
- 책 선택과 질문 작성
- 질문 유형 선택과 AI 결과 표시
- 기록 조회 및 질문 삭제
- 교사 피드백 등록
- AI 학생 진단과 HWPX 다운로드

테스트 계정이나 실제 데이터 접근이 필요한 항목은 사용 가능한 인증 상태 범위에서 확인하며, 인증 정보를 코드·문서·로그에 기록하지 않는다.

## 13. 스킬과 공식 문서 적용

구현 단계에서 최소한 다음 도구를 사용한다.

- `frontend-design`: 전체 시각 언어와 페이지 완성도
- `vercel-react-best-practices`: Next.js·React 성능과 번들 경계
- `web-design-guidelines`: 접근성 및 웹 UI 검토
- `imagegen`: 로우폴리 배경·아이콘 등 래스터 자산 생성
- Context7: Next.js 16, Mantine 9, Three.js, Motion의 최신 사용법 확인
- Superpowers: 구현 계획, 테스트 우선 개발, 검증 및 코드 리뷰 절차

3D 전용 커뮤니티 스킬은 출처와 품질을 검토한 뒤 필요할 때만 설치한다. 공식 Three.js 문서와 Context7 지침을 우선한다.

## 15. 완료 조건

1. 기존 기능과 데이터 흐름이 유지된다.
2. 모든 사용자 페이지가 B안의 로우폴리 디자인 시스템을 공유한다.
3. 모든 화면에 배경 모션 또는 적절한 정적 폴백이 존재한다.
4. 배경 때문에 텍스트, 버튼, 입력창, 차트가 가려지지 않는다.
5. 모션 감소, 모바일, WebGL 실패 환경에서 핵심 기능을 사용할 수 있다.
6. 정적 검사와 프로덕션 빌드가 통과한다.
7. GitHub `main`에 변경이 푸시된다.
8. Netlify `mokpo-imsung-v1`에 프로덕션 배포되고 실제 사이트를 검증한다.
